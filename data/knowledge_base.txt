# Machine Learning Fundamentals

## Three Types of Machine Learning
Machine learning is divided into three main paradigms:
1. **Supervised Learning** — models learn from labeled data (input-output pairs) to predict labels for unseen data. Examples: classification, regression.
2. **Unsupervised Learning** — models find patterns in unlabeled data. Examples: clustering, dimensionality reduction, anomaly detection.
3. **Reinforcement Learning** — an agent learns to make decisions by interacting with an environment and receiving rewards or penalties.

## Overfitting
Overfitting occurs when a machine learning model learns the training data too well, including its noise and outliers, at the expense of generalization to new data. The model memorizes training examples instead of learning underlying patterns. Symptoms include near-perfect training accuracy but poor validation/test performance. Prevention techniques include regularization, early stopping, cross-validation, and using more training data.

## Bias-Variance Tradeoff
- **Bias** measures systematic error — a model that consistently misbehaves in the same way (e.g., too simple to capture real patterns).
- **Variance** measures sensitivity to small fluctuations in training data — a model that changes wildly with different training sets (e.g., too complex).
The tradeoff: as model complexity increases, bias decreases but variance increases. The goal is to minimize total error = bias^2 + variance + irreducible noise.

## Regularization
Regularization prevents overfitting by adding a penalty term to the loss function that discourages overly complex models. Common forms:
- **L1 (Lasso)** — adds sum of absolute weights; promotes sparsity (feature selection).
- **L2 (Ridge)** — adds sum of squared weights; shrinks weights toward zero smoothly.
- **Dropout** — randomly deactivates neurons during training to prevent co-adaptation.
- **Elastic Net** — combines L1 and L2 penalties.

## Cross-Validation
Cross-validation (CV) is a model validation technique that splits data into k folds (commonly k=5 or 10). The model is trained k times, each time using k-1 folds for training and 1 fold for validation. The average performance across all folds gives a more robust estimate of generalization than a single train/test split. Stratified k-fold CV preserves class proportions in each fold.

# Neural Networks

## Backpropagation
Backpropagation is the algorithm used to train neural networks. It computes the gradient of the loss function with respect to each weight by applying the chain rule of calculus, propagating errors backward from the output layer to the input layer. The gradients are then used by an optimizer (e.g., SGD, Adam) to update weights in the direction that minimizes loss. It requires differentiable activation functions.

## Convolutional Neural Networks (CNNs)
CNNs are primarily used for **image recognition and computer vision tasks** such as image classification, object detection, and semantic segmentation. They use learnable filters (kernels) that slide across the input image to detect local features like edges, textures, and shapes. Key layers: convolutional layers, pooling layers, and fully connected layers.

## LSTM and Vanishing Gradient Problem
LSTMs (Long Short-Term Memory networks) address the vanishing gradient problem through a specially designed cell state with **gated mechanisms**:
- **Forget gate** — decides what information to discard from the cell state.
- **Input gate** — decides what new information to store.
- **Output gate** — decides what to output based on the cell state.
The cell state acts as a "highway" that allows gradients to flow unchanged across long sequences, preventing them from exponentially shrinking during backpropagation through time (BPTT).

## Activation Functions
Common activation functions in neural networks:
- **ReLU (Rectified Linear Unit)** — f(x) = max(0, x). Most widely used due to computational efficiency and non-saturation in positive region.
- **Sigmoid** — f(x) = 1/(1+e^-x). Output in (0,1), used for binary classification output layers.
- **Tanh** — f(x) = (e^x - e^-x)/(e^x + e^-x). Output in (-1,1), zero-centered, often used in hidden layers of RNNs.
- **Softmax** — normalizes output into a probability distribution over K classes.
- **Leaky ReLU** — f(x) = 0.01x for x<0, x for x>=0. Prevents "dying ReLU" problem.

## Pooling Layers in CNNs
Pooling layers reduce the spatial dimensions (width, height) of feature maps while retaining important features. Common types:
- **Max Pooling** — takes the maximum value in each pooling window. Extracts the most prominent feature.
- **Average Pooling** — takes the average value. Provides smooth feature summarization.
Pooling introduces translation invariance, reduces computation, and helps prevent overfitting.

# NLP and Transformers

## Transformer Architecture
The Transformer is a neural network architecture introduced in "Attention Is All You Need" (2017) that relies entirely on **self-attention mechanisms**, removing recurrence. Key components:
- **Encoder** — processes the input sequence in parallel; produces contextualized representations.
- **Decoder** — generates output autoregressively, attending to previous tokens and encoder output.
- **Positional Encoding** — adds positional information using sine/cosine functions since there is no recurrence.
- **Multi-Head Attention** — runs attention in parallel with multiple learned projection sets.

## Word Embeddings
Word embeddings are dense vector representations of words where similar words have similar vectors. They capture semantic meaning by projecting words into a continuous vector space. Examples: Word2Vec, GloVe, FastText. Words with similar contexts appear close in the embedding space, enabling arithmetic like "king - man + woman ≈ queen".

## Transfer Learning in NLP
Transfer learning in NLP involves pretraining a large language model on a massive general-purpose corpus, then fine-tuning it on a smaller downstream task dataset. Benefits: dramatic reduction in labeled data needed, faster training, and significantly better performance. Examples: BERT (pretrained on BooksCorpus + Wikipedia, fine-tuned for classification/NER/QA).

## BERT and Language Understanding
BERT (Bidirectional Encoder Representations from Transformers) handles language understanding by using a **bidirectional Transformer encoder** — unlike previous models that read text left-to-right or right-to-left independently, BERT reads both directions simultaneously. This allows each token to condition on its full context (left + right). Pretraining uses two tasks: Masked Language Modeling (MLM) and Next Sentence Prediction (NSP).

## Self-Attention
Self-attention computes the relevance of each token in a sequence to every other token, allowing the model to capture long-range dependencies regardless of distance. It computes: Attention(Q, K, V) = softmax(QK^T / sqrt(d_k)) * V, where Q (queries), K (keys), and V (values) are linear projections of the same input. "Multi-head" attention runs this mechanism in parallel multiple times with different projections.

## GPT and Its Training
GPT (Generative Pretrained Transformer) is a decoder-only Transformer trained in two stages:
1. **Pretraining** — language modeling objective: predict the next token given all previous tokens. Trained on large unannotated text corpora.
2. **Fine-tuning** — supervised fine-tuning on labeled task data (e.g., classification, QA) and/or RLHF (Reinforcement Learning from Human Feedback).
GPT-3 scaled this approach dramatically with 175B parameters and in-context learning (no fine-tuning needed for new tasks).

## Instruction Tuning
Instruction tuning is a fine-tuning technique where a pretrained language model is further trained on datasets of instruction-response pairs. This teaches the model to follow natural language instructions, improving zero-shot and few-shot task generalization. Unlike basic fine-tuning on a single task, instruction tuning aims to produce versatile assistants.

## RLHF (Reinforcement Learning from Human Feedback)
RLHF trains language models using human preference feedback:
1. Collect demonstration data → supervised fine-tuning (SFT) to produce an initial policy.
2. Have the current policy generate multiple responses per prompt → human raters rank responses by preference.
3. Train a reward model on ranking data → use it as a reward signal for Proximal Policy Optimization (PPO).
Result: models that are more helpful, harmless, and aligned with human values.

## Chain-of-Thought Prompting
Chain-of-thought (CoT) prompting encourages the LLM to generate intermediate reasoning steps before giving the final answer. This improves performance on complex multi-step reasoning tasks (math, logic, commonsense QA). Variants include few-shot CoT (providing examples with reasoning chains) and zero-shot CoT (prompting with "let's think step by step").

## GPT-4 Context Length
GPT-4 supports a context window of up to 128,000 tokens (depending on the specific model variant — GPT-4 Turbo and later versions extended the window). This allows processing very long documents, entire codebases, or extended conversations in a single prompt.

# RAG (Retrieval-Augmented Generation)

## What is RAG?
Retrieval-Augmented Generation (RAG) is an architecture that combines a retriever (finds relevant documents from a knowledge base) with a generator LLM. The retriever fetches relevant context, which is prepended to the user's question as a system prompt. The LLM then generates an answer grounded in the retrieved evidence, reducing hallucination and providing up-to-date knowledge.

## Main Components of a RAG Pipeline
1. **Data Loading** — ingest documents from various sources (PDFs, web, databases).
2. **Text Splitting / Chunking** — split documents into smaller pieces for efficient embedding.
3. **Embedding** — convert text chunks into dense vector representations.
4. **Vector Store / Index** — store embeddings in a vector database (FAISS, Pinecone, Chroma).
5. **Retriever** — given a query, fetch the k most similar chunks.
6. **Generator (LLM)** — combine retrieved context with the query to generate an answer.
7. **Optional: Rerankers, Hybrid Search, Query Rewriting**.

## Dense Retrieval
Dense retrieval uses neural network-based embedding models (e.g., BERT-based encoders) to represent queries and documents as dense vectors in a continuous embedding space. The retriever computes cosine similarity or dot product between the query vector and all document vectors. Unlike sparse retrieval (BM25), dense retrieval captures semantic similarity beyond keyword overlap.

## Chunking Strategy in RAG
Chunking strategy is critical because it determines the granularity of retrieved context. Small chunks may miss broader context; large chunks may include irrelevant noise. Considerations:
- **Chunk size** — larger chunks preserve more context but reduce precision.
- **Chunk overlap** — overlapping edges helps retain context cut off by splits.
- **Semantic chunking** — split by natural sentence or paragraph boundaries instead of fixed character counts.
- **Hierarchical chunking** — store both small chunks (for retrieval) and parent chunk summaries (for broader context).

## Advanced RAG Techniques
Beyond basic retrieval, advanced RAG techniques include:
- **Query rewriting** — use an LLM to reformulate the user query for better retrieval.
- **Reranking** — after initial retrieval, use a cross-encoder (e.g., Cohere rerank) to reorder results.
- **Hybrid search** — combine dense and sparse retrieval (FAISS + BM25).
- **Self-query retrieval** — extract metadata filters from the query (e.g., date, category).
- **Recursive chunking** — progressively split text to optimal granularity.
- **Contextual chunk embeddings** — prepend chunk-level metadata/summaries to each chunk embedding.

## Vector Databases
Vector databases are specialized storage systems optimized for storing and searching high-dimensional embedding vectors. They enable fast approximate nearest neighbor (ANN) search at scale. They store the embedded representations of documents and support similarity queries like "find the k most similar documents to this query vector." Examples: Pinecone, Weaviate, Chroma, Qdrant, Milvus.

## FAISS
FAISS (Facebook AI Similarity Search) is an open-source library developed by Meta for efficient dense vector similarity search and clustering. It provides algorithms for exact and approximate nearest neighbor search, supporting billions of vectors. It includes implementations of IVF (inverted file index), HNSW (hierarchical navigable small world), and PQ (product quantization) for fast, memory-efficient search.

## How Text Embeddings Capture Semantic Meaning
Text embeddings map textual data (words, sentences, documents) into dense numerical vectors in a continuous space. Similar texts cluster together — words used in similar contexts have similar vectors. These embeddings are learned by neural networks trained on large text corpora to predict context (Word2Vec), predict masked tokens (BERT), or optimize similarity objectives. The geometric properties of the space (angle, distance) encode semantic relationships.

## HNSW
HNSW (Hierarchical Navigable Small World) is an approximate nearest neighbor (ANN) algorithm used in vector databases. It builds a multi-layer graph where higher layers act as shortcuts for long-range navigation, and lower layers provide fine-grained local search. At query time, it navigates from the top layer downward using greedy routing. HNSW offers excellent speed-accuracy tradeoffs and is used by FAISS, Qdrant, and Weaviate.

## Hybrid Search
Hybrid search combines two or more retrieval methods, typically:
- **Dense retrieval** (embedding similarity) — captures semantic meaning.
- **Sparse retrieval** (BM25 / keyword matching) — captures exact term matches.
The results are merged using reciprocal rank fusion (RRF) or score weighted combination. Hybrid search addresses the limitation that pure dense search misses keyword-based matches and pure sparse search misses semantic matches.

# LangChain Ecosystem

## LangChain
LangChain is an open-source framework for developing applications powered by large language models. It provides modular abstractions for:
- **Model I/O** — standardized interfaces for prompts, LLMs, and output parsers.
- **Data connection** — document loaders, text splitters, vector stores, and retrievers.
- **Chains** — composing multi-step workflows combining LLMs with other components.
- **Agents** — enabling LLMs to reason and take actions (use tools).
- **Memory** — persisting conversation history across turns.
LangChain supports both Python and JavaScript/TypeScript.

## LangChain Expression Language (LCEL)
LCEL (LangChain Expression Language) is a declarative syntax for composing LangChain components using the pipe operator (|). It enables easy chaining of Runnable objects (retrievers, prompts, models, parsers) with automatic parallelization, retries, and streaming support. Example: `chain = ({"context": retriever | format_docs, "question": RunnablePassthrough()} | prompt | llm | parser)`. All LCEL chains are natively traceable by LangSmith.

## LangGraph
LangGraph is a library for building stateful, multi-actor applications with LLMs using a graph-based approach. It extends LangChain with:
- **Cycles** — allowing iterative refinement loops (e.g., self-correction, reflection).
- **Persistence** — checkpointing state to recover from failures.
- **Multi-agent orchestration** — defining workflows with multiple LLM agents communicating via graph edges.
It is designed for complex agentic workflows that require memory and human-in-the-loop checkpoints.

## LangChain Memory Types
LangChain supports several memory types for persisting conversation context:
- **ConversationBufferMemory** — stores all messages as a string or list.
- **ConversationBufferWindowMemory** — keeps only the last k messages.
- **ConversationTokenBufferMemory** — keeps messages within a token limit.
- **ConversationSummaryMemory** — summarizes past conversation to stay within token limits.
- **VectorStore-backed Memory** — stores memories as embeddings in a vector store for semantic retrieval.
- **Entity Memory** — tracks facts about entities mentioned in conversation.

## LangChain Retrievers
LangChain retrievers are interfaces that define how to fetch relevant documents from a data source. Built-in retrievers include:
- **VectorStore Retriever** — standard retriever backed by a vector database.
- **Self-Query Retriever** — translates natural language queries into metadata filters.
- **Time-Weighted Retriever** — prioritizes recently accessed documents.
- **Multi-Query Retriever** — generates multiple query variants using an LLM to improve recall.
- **Ensemble Retriever** — combines multiple retrievers with reciprocal rank fusion.
- **Parent Document Retriever** — retrieves at the chunk level but returns the parent document.

## LangSmith
LangSmith is a platform for debugging, evaluating, and monitoring LLM applications built with LangChain (or any LangGraph/LangChain project). It provides:
- **Traces** — step-by-step execution logs of every LLM call, retrieval, and chain invocation.
- **Datasets** — curated collections of inputs and reference outputs for evaluation.
- **Evaluators** — automated scoring of LLM outputs (faithfulness, answer relevancy, etc.).
- **Prompt Hub** — a repository for versioning, sharing, and deploying prompts across environments.
- **Monitoring** — latency, token usage, and quality metrics for production traces.

## LangSmith Traces
LangSmith traces capture detailed information about every LLM application execution:
- Input and output of each component (prompt, LLM call, tool execution, retrieval).
- Latency per step and total chain latency.
- Token usage and cost estimates.
- Metadata (user ID, session ID, tags).
- Exception traces and error details.
Traces are linked via a unique trace ID, allowing drill-down from a high-level view to individual LLM calls. Tracing is enabled by setting `LANGCHAIN_TRACING_V2=true` and providing `LANGCHAIN_API_KEY`.

## LangSmith Prompt Hub
The LangSmith Prompt Hub is a centralized repository for managing prompt templates. Features:
- **Versioning** — track changes to prompts over time with version history.
- **Collaboration** — share prompts across team members and environments (dev/staging/prod).
- **A/B Routing** — select between prompt variants for experiments.
- **Deployment** — publish approved prompts to production with a single update.
- **Metadata** — attach descriptions, tags, and example inputs to each prompt.
Prompts can be fetched by name/tag and used directly in LangChain chains.

## LangSmith for Production Monitoring
LangSmith helps monitor production LLM applications by:
- Collecting traces from live traffic to analyze latency distributions and error rates.
- Running evaluation datasets against production outputs to detect regressions.
- Setting up automatic alerts for quality drops (low faithfulness scores, high hallucination rates).
- Providing dashboards for token usage, cost, and latency across projects.
- Enabling A/B comparisons between prompt versions or model variants.

## LangSmith Datasets
LangSmith datasets are curated collections of examples used for evaluating LLM application quality. Each example contains an input (user query) and optionally reference answers (ground truth), contextual documents, or expected metadata. Datasets power automated evaluators (RAGAS, custom heuristics) to score production traces against benchmarks, enabling regression detection and quality monitoring over time.

# RAG Evaluation with RAGAS

## What is RAGAS?
RAGAS (RAG Assessment) is an open-source framework for evaluating Retrieval-Augmented Generation systems. It provides model-based metrics that assess quality without human reference labels, using an LLM as a judge. Metrics include faithfulness, answer relevancy, context precision, and context recall.

## RAGAS Faithfulness
Faithfulness measures how factually accurate the generated answer is relative to the retrieved context. It is computed by:
1. Extracting statements from the generated answer.
2. Using an LLM to judge whether each statement is supported by the retrieved context.
3. Faithfulness score = (number of supported statements) / (total number of statements).
A score of 1.0 means every claim in the answer is grounded in the retrieved documents. Low faithfulness indicates hallucination.

## Answer Relevancy in RAGAS
Answer relevancy measures how well the generated answer addresses the user's question. It is computed by:
1. Generating multiple potential questions from the answer using an LLM.
2. Computing the semantic similarity between the original question and the generated questions.
3. The score is the average cosine similarity — higher similarity means the answer directly addresses the question.
Low relevancy indicates the answer is off-topic, verbose, or only partially addresses the query.

## Context Recall in RAGAS
Context recall measures whether all relevant information from the ground truth answer is present in the retrieved context. It is computed by:
1. Splitting the ground truth answer into individual facts/statements.
2. Checking (via LLM) whether each ground truth statement is inferable from the retrieved context.
3. Recall = (number of inferable statements) / (total number of ground truth statements).
High context recall means the retriever found all the information needed to answer correctly.

## RAGAS Evaluation Inputs
RAGAS requires the following inputs to compute metrics:
- **user_input** — the original user question/query.
- **retrieved_contexts** — the list of documents/chunks retrieved by the RAG pipeline.
- **response** — the LLM-generated answer.
- **ground_truth** (optional, needed for context recall) — the reference answer.
RAGAS then uses an LLM (e.g., GPT-3.5 or GPT-4) as a judge to score each metric without requiring human annotation.

# Guardrails and AI Safety

## Guardrails AI
Guardrails AI (now part of NVIDIA NeMo Guardrails) is an open-source toolkit for adding safety and reliability guardrails to LLM-based applications. It allows developers to define rules that intercept, validate, and modify LLM inputs and outputs. Use cases: preventing PII leakage, blocking disallowed topics, ensuring structured output formats, and detecting jailbreak attempts.

## PII Detection
PII (Personally Identifiable Information) refers to any data that can identify an individual, such as names, email addresses, phone numbers, social security numbers, credit card numbers, and physical addresses. Detecting PII in LLM responses is important for:
- **Privacy compliance** — regulations like GDPR and CCPA require protecting PII.
- **Data sanitization** — redacting PII from logs and outputs before storage or display.
- **Hallucination detection** — a model generating fake PII can cause reputational or legal harm.
Guardrails AI uses NER (Named Entity Recognition) models to detect and redact PII entities.

## Structured Output Validation
Structured output validation ensures that an LLM's response conforms to an expected format or schema (e.g., JSON with specific fields, enum values within allowed ranges). This is critical for:
- Downstream system integration (parsers expecting specific JSON structure).
- Type safety in applications.
- Reducing errors from hallucinated or malformed outputs.
Guardrails AI uses JSON schema validation to check outputs, optionally with reasking the LLM to correct malformed responses.

## Constitutional AI
Constitutional AI (CAI) is a method for training AI systems to be helpful, harmless, and honest. It uses a set of guiding principles ("constitutions") to:
1. **Critique** — have the model identify harmful or unhelpful aspects of its own responses.
2. **Revise** — prompt the model to rewrite responses to align with constitutional principles.
3. **RLHF** — use these AI-generated critiques as preference data for reinforcement learning.
This reduces reliance on human labelers for harm identification and enables scalable alignment.

## Common AI Safety Concerns with LLMs
- **Hallucination** — LLMs generate plausible but factually incorrect statements.
- **PII leakage** — models may inadvertently expose sensitive information from training data.
- **Prompt injection** — adversarial inputs that manipulate model behavior.
- **Bias and stereotyping** — models reflect harmful biases present in training data.
- **Misinformation** — models generate convincing false content.
- **Lack of robustness** — small input perturbations cause large output changes.
- **Privacy violations** — models memorize and regurgitate training data containing personal information.
- **Over-reliance** — users trust model outputs without verification.
